# Write Operations — Implementation Spec

**Parent:** [v0.2.0](v0.2.0.md)
**Priority:** P0 — primary feature of v0.2.0
**Principle:** Let AI handle everything without leaving the assistant

---

## Why

Today, MCP Pool is read-only. When a user asks "resolve that Sentry issue" or "refund that Stripe charge", the AI has to say "I can't do that, go to the dashboard." That breaks the flow. Write operations close this gap — the AI becomes a full operator, not just a viewer.

---

## Design

### Always available — no flag needed

Write tools are **always registered** alongside read tools. No `ENABLE_WRITES` flag. The user installed the server because they want to use it — gating features behind flags adds friction without real safety.

Safety comes from the **MCP tool annotations** layer instead.

### User consent via MCP tool annotations

The MCP protocol (supported by SDK v1.29.0) provides `annotations` on each tool that tell the client how the tool behaves:

```typescript
{
  name: "resolve_issue",
  description: "Resolve a Sentry issue by ID",
  inputSchema: z.toJSONSchema(schema),
  annotations: {
    title: "Resolve Issue",
    readOnlyHint: false,      // this tool modifies state
    destructiveHint: false,   // not destructive (can be undone)
    idempotentHint: true,     // resolving twice is safe
  }
}
```

**How annotations drive consent:**

| Annotation | Value | Client behavior |
|------------|-------|-----------------|
| `readOnlyHint: true` | Read tools | Client auto-approves (current behavior) |
| `readOnlyHint: false` | Write tools | Client prompts user for confirmation |
| `destructiveHint: true` | Dangerous writes | Client shows stronger warning |
| `idempotentHint: true` | Safe to retry | Client may auto-approve retries |

MCP clients (Claude Desktop, Cursor, Claude Code) use these annotations to decide when to prompt the user. The server doesn't need to implement confirmation logic — the client handles it.

### Annotation categories for our write tools

| Category | `readOnlyHint` | `destructiveHint` | `idempotentHint` | Examples |
|----------|---------------|-------------------|-------------------|----------|
| Read | `true` | `false` | `true` | `get_issue`, `list_customers` |
| Safe write | `false` | `false` | `true` | `resolve_issue`, `assign_issue` |
| Create | `false` | `false` | `false` | `create_issue`, `create_contact` |
| Risky write | `false` | `true` | `false` | `cancel_order`, `send_email` |

### Also annotate existing read tools

While we're at it, add `readOnlyHint: true` to all existing read tools. This explicitly tells clients they're safe to auto-approve.

### Claude Desktop config — unchanged

No new env vars needed. Config stays exactly as it is today:

```json
{
  "mcpServers": {
    "stripe": {
      "command": "npx",
      "args": ["-y", "@vineethnkrishnan/stripe-mcp"],
      "env": {
        "STRIPE_SECRET_KEY": "sk_live_..."
      }
    }
  }
}
```

## File structure per package

```
packages/<name>/src/
├── tools/
│   ├── customer.tools.ts          # existing read tools
│   ├── customer.tools.test.ts
│   ├── customer.write-tools.ts    # NEW: write tools
│   └── customer.write-tools.test.ts
├── services/
│   └── <name>.service.ts          # add write methods here
└── index.ts                       # register both read and write schemas
```

**Convention:** Write tool files are `<entity>.write-tools.ts` — keeps reads and writes separated, easy to audit.

## Tool naming convention

Write tools use action verbs that make intent clear to the AI:

| Action | Prefix | Example |
|--------|--------|---------|
| Create new entity | `create_` | `create_issue`, `create_contact` |
| Update existing entity | `update_` | `update_issue_status`, `update_customer_metadata` |
| Execute action | verb | `resolve_issue`, `refund_charge`, `acknowledge_incident` |

Read tools keep existing names (`get_`, `list_`, `search_`).

## Implementation order

Phase 1 (highest user impact — start here):
1. **Sentry** — resolve/assign issues (most commonly requested action)
2. **Linear** — create issues, update status (project management flow)
3. **PagerDuty** — acknowledge/resolve incidents (on-call workflow)

Phase 2 (business operations):
4. **Stripe** — refunds, subscription management
5. **HubSpot** — create/update contacts and deals
6. **Intercom** — reply to conversations

Phase 3 (content & infrastructure):
7. **Notion** — create pages, append content
8. **Google Workspace** — calendar events, email drafts
9. **Shopify** — order/product management
10. **Datadog** — mute monitors, create downtime
11. **Vercel** — trigger deployments

## Shared patterns

### Tool schema with annotations

Write tool schemas include an `annotations` field alongside `description` and `schema`:

```typescript
// issue.write-tools.ts
export const IssueWriteToolSchemas = {
  resolve_issue: {
    description: "Resolve a Sentry issue, marking it as fixed.",
    schema: z.object({
      issue_id: z.string().describe("The Sentry issue ID (e.g., '12345')."),
    }),
    annotations: {
      title: "Resolve Issue",
      readOnlyHint: false,
      destructiveHint: false,
      idempotentHint: true,
    },
  },
  assign_issue: {
    description: "Assign a Sentry issue to a user or team.",
    schema: z.object({
      issue_id: z.string().describe("The Sentry issue ID."),
      assignee: z.string().describe("Username, email, or team slug to assign to."),
    }),
    annotations: {
      title: "Assign Issue",
      readOnlyHint: false,
      destructiveHint: false,
      idempotentHint: true,
    },
  },
};
```

### Updated ListTools handler

The `ListToolsRequestSchema` handler passes annotations through to the client:

```typescript
server.setRequestHandler(ListToolsRequestSchema, async () => {
  return {
    tools: Object.entries(AllToolSchemas).map(([name, config]) => ({
      name,
      description: config.description,
      inputSchema: z.toJSONSchema(config.schema),
      annotations: config.annotations ?? { readOnlyHint: true },
    })),
  };
});
```

**Note:** Existing read tools that don't have `annotations` default to `{ readOnlyHint: true }` — no changes needed to existing tool schemas.

### Write tool response format

Every write tool returns the updated/created resource so the AI can confirm the action:

```typescript
async resolve_issue(args: z.infer<typeof schema>) {
  const issue = await this.sentryService.resolveIssue(args.issue_id);
  return formatMcpResponse(issue, "Issue resolved successfully");
}
```

### formatMcpResponse enhancement

Extend the existing `formatMcpResponse` to accept an optional action message:

```typescript
export function formatMcpResponse(
  data: unknown,
  actionMessage?: string,
): { content: Array<{ type: string; text: string }> } {
  const result = actionMessage
    ? { _action: actionMessage, ...sanitize(data) }
    : sanitize(data);
  return {
    content: [{ type: "text", text: JSON.stringify(result, null, 2) }],
  };
}
```

### Error handling for write operations

Write operations should return clear, actionable error messages:

```typescript
// Bad: "Error: 400 Bad Request"
// Good: "Error: Cannot refund charge ch_123 — charge has already been refunded."
```

## What's NOT in scope for v0.2.0

- **Bulk operations** — no batch creates/updates
- **Hard deletes** — archive/soft-delete only (except calendar events which are inherently deletable)
- **Billing mutations** — no plan changes, price creation
- **Permission management** — no user/team/role changes
- **File uploads** — no attachment handling

These are candidates for v0.3.0 after the write pattern is proven.

---

## Testing strategy

Each write tool gets:
1. **Unit test** — mock the service, verify tool calls service with correct args and returns formatted response
2. **Schema test** — verify Zod schema validates correct inputs and rejects bad inputs
3. **Error test** — verify error handling for common failure modes (not found, permission denied, conflict)

Same pattern as existing read tool tests, but with additional focus on input validation since writes are more sensitive.
